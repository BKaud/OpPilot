# Database Configuration

## Overview

The database configuration uses a simple file-based approach to determine whether to use **development** or **production** credentials.

## How It Works

**Priority Order (first match wins):**

1. **Production**: If `DBfiles/app_config.php` exists → Use production config
2. **Local Override**: If `DBfiles/db_local_config.php` exists → Use your personal config (**NOT tracked by Git**)
3. **Development Defaults**: Use hardcoded defaults in `db_config.php`

### Production Environment
- **When**: `app_config.php` EXISTS
- **Configuration**: Auto-generated by GitHub Actions deployment
- **Database Credentials**: Loaded from GitHub Secrets
- **Important**: Since your DB is hosted on the same server as the frontend, set `SERVER_HOST` secret to `localhost` or `127.0.0.1`

### Local Override (Keep Passwords Out of Git)
- **When**: `db_local_config.php` EXISTS (and no `app_config.php`)
- **Configuration**: Your personal config file (gitignored)
- **Use Case**: Hardcode your actual passwords without exposing them in the repo
- **Setup**:
  1. Copy the template: `cp db_local_config.php.template db_local_config.php`
  2. Edit `db_local_config.php` with your real credentials
  3. This file is gitignored - your passwords stay private!

### Development Defaults
- **When**: Neither `app_config.php` nor `db_local_config.php` exist
- **Configuration**: Hardcoded in `db_config.php`
- **Database Credentials**: Safe default values that can be committed to Git
- **Use Case**: Quick start for new developers or CI/CD environments

## Files

- **`db_config.php`**: Main configuration file with connection helper functions
- **`db_config_debug.php`**: Debug endpoint (localhost only) to verify configuration and test connectivity
- **`app_config.php`**: Production config (auto-generated by deployment, not in Git, presence triggers production mode)

## Testing Your Configuration

Visit: `http://localhost/OpPilot/DBfiles/db_config_debug.php`

This will show:
- Detected environment (development/production)
- Database host and connection details
- Connection test results

## Modifying Development Credentials

### Option 1: Local Override File (Recommended - Keeps Secrets Out of Git)

Create `DBfiles/db_local_config.php`:

```bash
# Copy the template
cp DBfiles/db_local_config.php.template DBfiles/db_local_config.php

# Edit with your real credentials
notepad DBfiles/db_local_config.php
```

Your `db_local_config.php`:
```php
<?php
define('BASE_PATH', '');
define('DB_HOST', 'localhost');
define('DB_PORT', 3306);
define('DB_USER', 'my_username');
define('DB_PASS', 'my_secret_password');  // Safe! Not tracked by Git
define('DB_NAME', 'oppilot');
```

### Option 2: Edit Defaults in db_config.php (Everyone Sees These)

Edit the development defaults section in `db_config.php`:

```php
} else {
    // === DEVELOPMENT DEFAULTS ===
    define('DB_HOST', 'your-dev-host');
    define('DB_PORT', 3306);
    define('DB_USER', 'your-dev-user');
    define('DB_PASS', 'your-dev-password');
    define('DB_NAME', 'your-dev-database');
}
```

⚠️ **Warning**: These values are committed to Git!

## Modifying Production Credentials

Update the GitHub Secrets in your repository settings:

- **`SERVER_HOST`**: Set to `localhost` or `127.0.0.1` (since DB is on same server as frontend)
- **`SERVER_USER`**: MySQL username for production
- **`SERVER_PASS`**: MySQL password for production
- **`SERVER_DB_NAME`**: Database name for production

The deployment workflow will automatically generate `app_config.php` with these values, which triggers production mode.

## Migration from Old System

The old system used complex environment detection and multiple fallback layers. The new system is dead simple:

**Old:** Check localhost IPs → Check XAMPP path → Check hostname → Check for app_config.php → Multiple fallbacks

**New:** Does `app_config.php` exist? Yes = Production, No = Development

## Troubleshooting

**Production using wrong credentials:**
- Check that `app_config.php` exists in production
- Verify GitHub Secrets are set correctly (especially `SERVER_HOST` should be `localhost`)

**Development using production credentials:**
- Delete `app_config.php` from your local machine (it shouldn't be there)
- This file is .gitignored and only exists on production server

**Can't connect in production:**
- If DB is on same server, ensure `SERVER_HOST` secret is set to `localhost` not a remote IP
- Check that MySQL allows connections from localhost in production
